# 5. 브랜치


지금 상황(Next.js + GitHub + Vercel 배포) 기준으로 
가장 많이 쓰이는 브랜치 전략 두 가지를 비교하고, 현재 프로젝트에 맞는 추천 전략을 정리해드릴게요.

---

# 🔹 1. GitHub Flow (간단, Vercel 연동에 적합)

* **master (또는 main)**
  → 항상 배포 가능한 상태
  → Vercel Production 배포가 자동 연결됨

* **feature/** 브랜치
  → 새로운 기능이나 수정 사항은 `feature/기능명` 으로 분기
  → 작업 후 → GitHub Pull Request(PR) → master에 merge

### 예시

```
master
 └─ feature/section2-image
 └─ feature/header-nav
```

Vercel은 **PR을 올리면 Preview URL 자동 생성** → 테스트 후 merge → Production 자동 배포

---

# 🔹 2. Git Flow (복잡, 대규모 팀 프로젝트에 적합)

* **master**    → 실제 배포 (Production)
* **develop**   → 개발 통합 브랜치
* **feature/**  → 기능 개발 (develop에서 분기)
* **release/**  → 배포 준비 (develop에서 분기 → master merge)
* **hotfix/**   → 운영 중 긴급 수정 (master에서 직접 분기)

### 예시

```
master
 └─ hotfix/security
develop
 └─ feature/section2-image
 └─ release/v1.0.0
```

규모가 크고, QA/테스트 과정이 별도로 있는 경우 적합

---

# 🔹 지금 프로젝트에 추천하는 전략

**GitHub Flow**
이유:

* 개인/소규모 프로젝트
* Vercel은 **master = Production, branch/PR = Preview** 자동화 구조와 딱 맞음
* 굳이 develop, release 같은 중간 단계를 두지 않아도 됨

---

# ✅ 실제 적용 방법 (GitHub Flow)

1. **새 작업 시작**
   ```bash
   git checkout -b feature/section2-image
   ```

2. **작업 후 커밋 & 푸시**
   ```bash
   git add .
   git commit -m "섹션2 이미지 작은 새로 교체"
   git push origin feature/section2-image
   ```

3. **GitHub Pull Request 생성**
   * base: master
   * compare: feature/section2-image
   * PR 생성 시 Vercel Preview 자동 생성

4. **테스트 후 merge**
   * Vercel Preview에서 확인
   * 이상 없으면 GitHub에서 Merge PR → master 반영
   * Vercel이 master 기준 Production 자동 배포

# 3. 패턴으로 대량 삭제 (고급)

특정 접두사(test-)가 붙은 원격 브랜치 여러 개 삭제:
```BASH
git branch -r | grep "origin/test-" | sed 's/origin\///' | xargs git push origin --delete
```

# 정리
로컬: git branch -d 또는 -D (여러 개 한 줄 가능)
원격: git push origin --delete (여러 개 한 줄 가능)



1) 원격 브랜치 목록을 보여줍니다.

```BASH
git branch -r
origin/master
origin/next
origin/test-1
origin/test-2
origin/feature-login
```

2) 목록에서 "origin/test-" 로 시작하는 줄만 골라냅니다.
```BASH
grep "origin/test-"

origin/test-1
origin/test-2
```

3) 
origin/브랜치명 → 브랜치명 으로 바꿔줍니다. (origin/ 제거)
```BASH
sed 's/origin\///'
test-1
test-2
```

4) 위에서 걸러낸 브랜치 이름들을 git push origin --delete 뒤에 붙여 실행합니다.

```BASH
git push origin --delete

git push origin --delete test-1 test-2

git push origin --delete future/babo
git push origin --delete future/moon 
git push origin --delete future/jong future/seon
git push origin --delete future
git push origin --delete develop/moon
git push origin --delete develop/soen
git push origin --delete develop/jong develop/babo develop/moon
git push origin --delete jong moon moonjong seon
```

즉, 한 번에 여러 원격 브랜치를 삭제하게 됩니다.
# 정리
이 명령어는:
원격 브랜치 목록 보기
이름이 origin/test- 로 시작하는 것만 찾기
origin/ 접두사 제거

```BASH
git push origin --delete  # 모두 삭제
```

# 사용 예시:
만약 원격에 origin/test-1, origin/test-2, origin/test-old 
브랜치가 있다면 → 한 번에 모두 삭제됩니다.



# 🔹 단계별 방법
1) 원격 브랜치 목록 확인
```BASH
git branch -r

origin/HEAD -> origin/master
origin/master
origin/feature/test
origin/feature/login
origin/dev
```

2) master 제외하고 모두 삭제
git branch -r | grep 'origin/' | grep -v 'origin/master' | sed 's/origin\///' | xargs -I {} git push origin --delete {}

# 명령어 설명

git branch -r → 원격 브랜치 목록 출력
grep 'origin/' → origin 브랜치만 선택
grep -v 'origin/master' → master 제외
sed 's/origin\///' → origin/브랜치명 → 브랜치명 으로 변경
xargs -I {} → 하나씩 git push origin --delete {} 실행
# 🔹 실행 전 안전하게 확인하기

# 삭제하기 전에 먼저 어떤 브랜치가 삭제될지 미리 확인해보려면:
```BASH

feature/login
dev
```

# 이렇게 출력되는 브랜치들이 실제로 삭제될 대상입니다.

# 정리

# SSH Public Key 만들기(완전 가이드) 
# 새 키 발급
# 1. 키 생성 (권장: 전용 파일명 사용)
```BASH
ssh-keygen -t rsa -b 4096 -C "nextjs_project" -f ~/.ssh/next
ssh-keygen -t rsa -b 4096 -C "nextapp_project" -f ~/.ssh/nextapp
ssh-keygen -t rsa -b 4096 -C "moonjong_ssh_key" -f ~/.ssh/moonjong
```
# 자세한 설명
ssh-keygen          ? SSH 키 생성 명령어
-t rsa              ? RSA 알고리즘 사용 (권장: ed25519)
-b 4096             ? 키 길이 (4096bit → 더 안전)
-C "nextjs_project" ? 키에 붙는 주석 (보통 프로젝트 이름/이메일)
-f ~/.ssh/next      ? 저장할 파일 이름(키이름) 지정

# 퍼블릭 키 구분(개인키, 공개키)
next            ← 개인키 (private key) (절대 공유 X)
next.pub        ← 공개키 (public key, 카페24에 등록한 것) 공개키 (서버나 GitHub에 등록)

nextapp         ← 개인키 (private key) (절대 공유 X)
nextapp.pub     ← 공개키 (public key, 카페24에 등록한 것) 공개키 (서버나 GitHub에 등록)

moonjong        ← 개인키 (private key) (절대 공유 X)
moonjong.pub    ← 공개키 (public key, 카페24에 등록한 것) 공개키 (서버나 GitHub에 등록)

# 9 Git 원격(remote) 등록 & 푸시
- 카페24 앱 리스트의 저장소 형식을 그대로 사용합니다. (보통 .git 확장자 없이)
```BASH
git remote remove cafe24 2>/dev/null
git remote add cafe24 moonjong22@next.cafe24app.com:moonjong22_next
git remote -v
# cafe24  moonjong22@next.cafe24app.com:moonjong22_next (fetch)
# cafe24  moonjong22@next.cafe24app.com:moonjong22_next (push)

# 첫 푸시
git push cafe24 master
```


# 10. Git 문제 해결 자주 쓰는 명령
```BASH
git reflog : 모든 커밋/리셋/체크아웃 이력 보기 (복구 가능)
git stash : 현재 작업중인 변경사항 임시 저장
git stash pop : 임시 저장된 변경사항 복원
git diff : 변경된 부분 비교
git blame 파일명 : 특정 파일의 각 줄을 누가, 언제 수정했는지 확인
```
